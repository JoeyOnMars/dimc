# 架构决策记录 (ADR): Dimcause 存储方案评估

**日期**: 2026-02-14
**状态**: ✅ 已决策 (保留现有架构)
**决策者**: Dimcause 架构组
**相关任务**: Phase 4.0

## 1. 背景 (Context)
Dimcause 作为一个本地优先 (Local-First) 的多智能体系统，需要维护一个因果图谱（Causal Graph）。目前的架构是 **NetworkX (In-Memory)** + **SQLite (Persistence)**。
我们需要评估是否需要迁移到专业的图数据库（如 Kùzu），以应对未来的扩展性需求。

## 2. 评估方案 (Alternatives)

### 方案 A: NetworkX + SQLite Registry (现状)
- **机制**: 数据持久化在 SQLite 表 (`graph_nodes`, `graph_edges`)，启动时全量加载到 NetworkX 对象。
- **优点**: 无额外依赖（纯 Python + Stdlib），极简架构，NetworkX 算法生态丰富。
- **担忧**: 加载速度（Hydration Time）和内存占用。

### 方案 B: Kùzu (嵌入式图数据库)
- **机制**: 磁盘即数据，支持 Cypher 查询语言，针对图遍历优化。
- **优点**: 处理亿级节点无压力，支持复杂模式匹配。
- **缺点**: 引入二进制依赖 (Wheels)，增加构建和分发复杂度，API 与 NetworkX 不兼容。

## 3. 基准测试结果 (Benchmark Results)
我们在 M4 Mac Mini 上对方案 A 进行了基准测试 (Python 3.13, NetworkX 3.4.2)。

| 规模 (Nodes) | 边数 (Edges) | 写入时间 (Batch SQL) | 加载时间 (Hydration) | 遍历时间 (BFS 2-hop) |
|-------------:|-------------:|---------------------:|---------------------:|---------------------:|
| **1,000**    | 1,979        | 0.006s               | 0.005s               | 0.005ms              |
| **10,000**   | 19,916       | 0.050s               | 0.047s               | 0.003ms              |
| **50,000**   | 100,213      | 0.287s               | 0.275s               | 0.004ms              |

**分析**:
1.  **加载速度**: 即使是 50k 节点（远超目前 Dimcause 的预期规模），加载时间也仅需 **0.28秒**。这对于 CLI 工具来说完全可以接受。
2.  **遍历速度**: 内存中遍历极快 (微秒级)，NetworkX 的性能完全不是瓶颈。
3.  **开发体验**: NetworkX 的 API 对 Python 开发者最友好，且支持所有主流图算法。

## 4. 决策 (Decision)
**决定保留 NetworkX + SQLite 架构，暂不引入 Kùzu。**

### 理由
1.  **性能足够**: 当前架构在 50k 节点规模下依然保持亚秒级响应，满足未来 1-2 年的需求。
2.  **架构极简**: 引入 Kùzu 会显著增加复杂度和包体积，收益（针对百万级节点的优化）在当前阶段无法体现。
3.  **生态兼容**: NetworkX 是标准，任何迁移都会导致重写大量算法逻辑。

## 5. 后续建议 (Recommendations)
1.  **优化加载**: 如果未来节点数超过 100k，可考虑实现按需加载 (Lazy Loading) 或分页查询。
2.  **JSON-LD 互通**: Phase 4.1 应专注于实现 JSON-LD 导出，以便与其他图工具互通，而不是更换底层存储。
