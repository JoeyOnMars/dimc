# 审计文档说明

## 定位

1. 本目录承载本项目的审计工件、对齐检查与事故复盘。
2. 本目录不是产品架构入口，不是状态入口，也不是 roadmap 入口。
3. 当前正式入口仍以 `README.md`、`docs/ARCHITECTURE_INDEX.md`、`docs/STATUS.md`、`docs/dev/BACKLOG.md`、`docs/dev/V6.0_ROADMAP.md` 为准。

## 目录结构

1. `postmortems/`
   - 事故复盘。
   - 作用是沉淀教训、反哺治理规则。
   - 可引用其教训，不应直接拿它判定当前代码现状。
2. `alignment/`
   - 文档与实现对齐审计。
   - 只在结论仍与 live 代码一致时可引用。
   - 引用前必须复核相关代码或正式文档。

## 使用规则

1. 当前代码现状，以 live 代码、测试结果和正式共享文档为准。
2. 审计文档不能反向改写产品定义、项目落地或仓库治理的正式文档。
3. 任何带时间性的审计结论，在复用前都必须重新核查代码和测试。
4. 历史审计材料不再保留在当前树中，默认通过 git history 追溯。

## 命名规则

1. 统一使用 `YYYY-MM-DD_<topic>.md`。
2. 历史遗留命名已在本轮重构中标准化。

## 当前保留策略

1. `postmortems/` 保留仍有治理价值的事故复盘。
2. `alignment/` 仅保留仍可作为历史对齐参考的审计。
3. 其余历史材料不再保留在当前树中，统一交给 git history 追溯。
