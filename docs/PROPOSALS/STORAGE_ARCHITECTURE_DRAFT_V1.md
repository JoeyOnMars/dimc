# 存储架构草案 v1

## 文档定位

1. 本文是**产品职责层**的《存储架构草案 v1》，目标是先把产品级存储职责、层间边界和生命周期说清楚，再为后续的 workspace profile 与实现迁移提供上位约束。
2. 本文**不是**当前仓库目录结构说明，不是任何具体 workspace 的开发流程规则汇总，也不是 `docs/STORAGE_ARCHITECTURE.md` 的直接改写稿。
3. 本文刻意不从任何当前 workspace 路径、当前起草流程或当前工程治理细节出发反推产品架构；这些内容若需保留，只能进入附录或后续 `workspace profile` 文档。
4. 本文正文只保留产品级事实、产品级推测和产品级存储约束；起草约束、审计溯源与过程性元信息统一放入文末附录。

## 推测与假设

1. [推测] 《存储架构草案 v1》的核心工作不应是选具体数据库、目录名或表结构，而应先定义四层职责、层间关系、真理源边界和生命周期。原因：系统重构文档明确说四层差异首先在职责与语义，不在介质。[基于事实 #6, #7]
2. [推测] `Relation Candidate`、验证后的关系、被否决关系，以及它们的证据绑定，都应进入 Knowledge Layer，而不是放到 Derived Index Layer。原因：产品主链和因果调查产品价值都要求这些结构成为可解释、可查询、可追溯的长期对象，而非可丢弃缓存。[基于事实 #3, #6, #9]
3. [推测] 报告、检查结果、调查附件等一旦形成用户可引用或可审计的产物，应归入 Evidence Layer；对应的运行中间状态、活跃进程与恢复点应归入 Runtime Layer。原因：系统重构文档已经按“可审计工件”与“高频可变状态”区分二者。[基于事实 #6, #7]
4. [推测] `workspace/default profile` 后续很可能会采用本地文件系统 + 本地数据库 + 本地索引的组合，但这种组合只能被视为某种默认实现，不应回写成产品级唯一答案。原因：产品是 local-first，但系统重构同时要求产品架构与具体 workspace 实现分离。[基于事实 #5, #8]
5. [推测] 当前尚不足以在本草案中冻结所有稳定 ID 规则、每层物理介质与精确 schema；当前事实层足以定义职责约束，但不足以直接定义对象合同与实现结构。原因：现有事实已经给出产品主链、四层职责和分级体系，但还没有把对象字段、版本合同和 profile 介质映射固定下来。[基于事实 #3, #4, #5, #6, #7, #8]

## 结论与建议

### 1. 总体判断

1. 产品级存储架构应被定义为：**为证据驱动因果调查系统提供四层职责分离的数据底座，使系统能够在不混淆证据、运行状态、结构化知识和派生索引的前提下完成调查、解释、回放与扩展。** 依据: 事实 #1, #2, #5, #6, #7。
2. 存储架构的主语必须是**产品职责**，不是任何当前 workspace 的路径、目录、流程门禁或治理习惯。否则会把“当前怎么开发”误写成“产品是什么”。依据: 事实 #5, #7, #8。
3. 本草案应先定义四层职责与边界，再把 default profile 与具体 workspace 映射放到后续文档。当前不应反过来从既有路径和实现习惯倒推四层。依据: 事实 #5, #6, #8, + 推测 #1, #4。

### 2. 存储架构的定义原则

1. **按职责分层，不按介质分层。** 同一种物理介质可以承载不同层，不同层也可以落在同一种数据库里；但职责、真理源地位和生命周期必须先分清。依据: 事实 #6, #7。
2. **按真理源地位分层。** 不是所有数据都同等重要；证据与结构化知识比派生索引更接近产品真理源，而运行状态则服务于“当前怎么跑”。依据: 事实 #6, #7。
3. **按可变性分层。** 高频变更的运行状态，不能与应长期稳定可审计的证据、报告和知识对象混在一起。依据: 事实 #6, #7, + 推测 #3。
4. **按生命周期分层。** 某些数据面向长期保留与回放，某些数据面向恢复和清理，某些数据随时可重建；这三种寿命模型必须显式区分。依据: 事实 #6, #7。
5. **按对外语义分层。** 用户引用的调查结果、证据包、报告和对象关系，不应与内部检索缓存或索引优化结构混为一谈。依据: 事实 #2, #6, #9。
6. **按产品主链服务。** 四层必须共同服务 `Material -> Object -> Relation Candidate -> Validation -> Causal Graph -> Explanation`，而不是围绕单一实现模块自我闭环。依据: 事实 #3, #5。

### 3. 产品级四层模型

#### 3.1 Evidence Layer

1. **定义**：Evidence Layer 是产品对“可被引用、可被审计、可被回放的证据载体”的持久记录层，是调查能被回看、被引用、被解释的基础。依据: 事实 #2, #6, #7。
2. **Evidence Layer 内部至少应区分两类子对象**：
   - `Source Materials`
   - `Generated Evidence Artifacts`
   依据: 事实 #2, #6, + 推测 #3。
3. **子对象 A：Source Materials**
   - 定义：指原始导入材料、原始文件及其版本快照、原始外部导出、原始输入快照。
   - provenance：其来源应回到外部材料源、导入批次、采集时间或快照边界。
   - 生命周期：默认采用最保守的证据保留语义，以保证后续调查仍能回到原始输入。
   - 版本语义：版本应围绕“材料何时被采到、何时形成快照、何时发生版本变化”定义，而不是围绕后续解释结论定义。
   - 引用方式：后续解释与报告应能稳定引用到具体材料、具体版本、具体片段或具体快照。
   依据: 事实 #2, #6, #7。
4. **子对象 B：Generated Evidence Artifacts**
   - 定义：指报告、检查结果、调查附件、用户可引用工件，以及需要回放或审计的系统生成产物。
   - provenance：其来源应回到生成它的 run、生成策略、输入范围、生成时间，以及必要时的人工确认动作。
   - 生命周期：默认共享 Evidence Layer 的长期保留、可审计、可引用语义；这是有意决策，因为这些产物同样会成为后续调查的证据输入，而不只是临时输出。
   - 版本语义：版本应围绕“哪次运行生成、是否被修订、修订依据是什么”定义，而不是沿用原始材料的版本编号。
   - 引用方式：后续解释与报告应能稳定引用到具体工件、具体生成批次、具体 run 以及工件内部可定位片段。
   依据: 事实 #2, #6, #7, + 推测 #3。
5. **两类子对象的共同点与差异**
   - 共同点：二者都属于证据载体，都必须保留 provenance，都必须可被稳定引用，都不应退化为只为性能服务的临时缓存。
   - 差异：`Source Materials` 代表外部世界被导入时的材料事实，`Generated Evidence Artifacts` 代表系统运行后产生的证据化工件；两者共享 Evidence Layer 的审计语义，但不共享同一套 provenance 与版本语义。
   依据: 事实 #2, #6, #7。
6. **不应存内容**：
   - 活跃运行状态
   - 进程句柄、锁、恢复点
   - 只为性能服务的缓存
   - 仅用于召回优化的索引副本
   依据: 事实 #6, #7。
7. **可变性**：倾向不可变或追加式写入；允许补充元数据和新增版本，但不应把 Evidence Layer 当作随时覆盖的状态表。依据: 事实 #6, #7。
8. **真理源地位**：它是“证据世界”的真理源之一。用户最终看到的“这条结论引用了哪些材料、哪些报告、哪些工件”，应能回到这一层。依据: 事实 #2, #9。
9. **生命周期**：长期保留、可归档、可审计、可回放；清理策略应保守，默认比 Runtime Layer 更长。`Source Materials` 与 `Generated Evidence Artifacts` 在证据保留级别上有意共享这套生命周期约束，但保留各自不同的 provenance 与版本语义。依据: 事实 #2, #6, #7。
10. **对外语义**：Evidence Layer 回答的是“你依据了什么”。它不是解释本身，也不是当前运行控制面。依据: 事实 #2, #6。

#### 3.2 Runtime Layer

1. **定义**：Runtime Layer 是产品对“运行进行中状态”的持久与半持久控制层，服务于创建、执行、暂停、恢复、检查、清理与失败恢复。依据: 事实 #5, #6。
2. **应存内容**：
   - `Run`、`RunSpec`、`RunState`
   - 状态迁移记录
   - 活跃执行上下文
   - 资源绑定、恢复点、生命周期标记
   - 当前运行与输入输出的绑定关系
   - inspect / resume / cleanup 所需状态
   依据: 事实 #5, #6。
3. **不应存内容**：
   - 作为长期知识真理源的对象关系
   - 用户长期引用的报告与证据包
   - 仅为召回性能存在的索引结构
   依据: 事实 #6, #7。
4. **可变性**：高频更新、并发敏感、状态机驱动；它必须被设计成可以恢复，也必须被设计成可以清理。依据: 事实 #5, #6, #7。
5. **真理源地位**：它是“当前系统怎么跑”的真理源，不是“世界上发生了什么”的最终真理源。依据: 事实 #6。
6. **生命周期**：与 run 生命周期强绑定；活跃数据短寿，历史状态可按恢复与审计需求保留摘要，但不应替代 Evidence Layer。依据: 事实 #5, #6, + 推测 #3。
7. **对外语义**：Runtime Layer 回答的是“系统现在在做什么、能否恢复、为何失败”。它不应被用户误当成证据层或知识层。依据: 事实 #5, #6。

#### 3.3 Knowledge Layer

1. **定义**：Knowledge Layer 是产品对“对象、关系、证据绑定与因果分级”的结构化建模层，是解释与查询的主语义层。依据: 事实 #2, #3, #6, #9。
2. **应存内容**：
   - `Material` 提升后的对象
   - 对象属性与稳定标识
   - 显式关系、候选关系、验证后关系、被否决关系
   - 对象到证据的绑定
   - 对象到运行的桥接
   - 关系状态历史
   - 因果覆盖等级与确定性等级
   - 解释所需的证据路径元数据
   依据: 事实 #3, #4, #6, + 推测 #2。
3. **不应存内容**：
   - 原始材料全文及其主副本
   - 活跃进程、锁和恢复点
   - 仅为加速查询存在的向量、FTS 或 traversal cache
   依据: 事实 #6, #7。
4. **可变性**：允许演化，但应保持对象与关系的身份稳定、状态可追溯；不应把它退化成无版本、无来源、可随意覆盖的临时表。依据: 事实 #3, #4, #6, #7。
5. **真理源地位**：它是“系统对材料世界做出的结构化建模”的真理源。用户查询“哪些对象存在、哪些关系成立、证据等级是什么”时，应以这一层为主。依据: 事实 #2, #3, #4, #9。
6. **生命周期**：比 Runtime 长，通常与调查知识资产同寿；可随着新证据和新 run 演化，但不应因索引重建而消失。依据: 事实 #6, #7。
7. **对外语义**：Knowledge Layer 回答的是“系统目前认为哪些对象和关系成立，以及为什么”。它既不是原始证据，也不是单纯缓存。依据: 事实 #2, #6, #9。
8. **关键限制**：Knowledge Layer 不能把一切都折叠成 `Event`，也不能只保存最终“确认边”。候选边、被否决边和证据缺口同样是因果调查系统的重要知识。依据: 事实 #3, #4, + 推测 #2。
9. **关系状态历史约束**：关系从候选、支持、确认到否决等状态迁移，必须保留可回放的历史；“当前状态”只能是某一时点上的投影，不得成为唯一记录。后续系统需要审计关系演化、比较不同时点的判断，并支持反事实复盘。依据: 事实 #3, #4, #6, #9, + 推测 #2。

#### 3.4 Derived Index Layer

1. **定义**：Derived Index Layer 是产品对“可重建索引与加速结构”的容器层，服务于召回、遍历、排序和查询性能。依据: 事实 #6, #7。
2. **应存内容**：
   - 向量索引
   - 全文索引
   - 图遍历优化结构
   - 代码与符号索引
   - 查询缓存
   - 其他明确标记为可重建的派生结构
   依据: 事实 #6。
3. **不应存内容**：
   - 唯一的对象与关系真理记录
   - 唯一的原始材料主本
   - 唯一的 run 状态记录
   依据: 事实 #6, #7。
4. **可变性**：允许失效、替换、重建、清空；其恢复代价应低于 Evidence 与 Knowledge 的丢失代价。依据: 事实 #6, #7。
5. **真理源地位**：它不是产品真理源，最多是性能与召回的增强器。依据: 事实 #6, #7, #9。
6. **生命周期**：由重建策略与性能需求决定，可以短于其它三层。依据: 事实 #7。
7. **对外语义**：Derived Index Layer 回答的是“怎样更快找到”，不是“什么是真实证据”或“哪条关系成立”。依据: 事实 #6, #9。

### 4. 层间关系与最小数据流

1. **导入**：材料首先进入 Evidence Layer 形成可引用的材料记录；Runtime Layer 记录这次导入 run 的状态；Knowledge Layer 从材料中抽取对象与关系；Derived Index Layer 为检索和遍历建立可重建索引。依据: 事实 #3, #5, #6。
2. **调查**：调查 run 在 Runtime Layer 中推进；它读取 Evidence 与 Knowledge，生成新的候选关系、验证结果与证据化工件；新的可引用工件写回 Evidence，结构化结论与关系状态历史写入 Knowledge。依据: 事实 #2, #3, #4, #6, + 推测 #2, #3。
3. **解释**：解释接口优先读取 Knowledge 中的对象、关系、等级与证据绑定，再回到 Evidence 取具体证据片段和工件；Derived Index 只辅助召回，不单独产出最终结论。依据: 事实 #3, #6, #9。
4. **重建**：Derived Index 应能由 Evidence 与 Knowledge 重建；Runtime 不应成为重建 Evidence 或 Knowledge 的唯一来源。依据: 事实 #6, #7。
5. **清理**：Runtime 可在 run 完结后主动收缩；Derived Index 可按重建策略淘汰；Evidence 与 Knowledge 的清理应更保守，并受保留策略控制。依据: 事实 #6, #7。

### 5. 层间写入边界

| 层 | 主要写入者 | 允许的主要写入类型 | 不应承担的职责 |
| --- | --- | --- | --- |
| Evidence | 导入器、报告生成器、检查器、人工补充记录 | 追加 `Source Materials`、`Generated Evidence Artifacts`、证据片段、附件 | 活跃 run 状态、仅为性能而存在的缓存 |
| Runtime | Runtime kernel、调度与恢复逻辑 | run 创建、状态迁移、恢复点、资源绑定 | 长期知识真理、最终证据档案 |
| Knowledge | 抽取器、验证器、因果推理器、人工校正流程 | 对象、关系、关系状态、关系状态历史、等级、证据绑定 | 原始材料主副本、纯性能索引 |
| Derived Index | 索引器、重建任务 | 向量、FTS、图遍历优化、缓存 | 唯一事实来源 |

依据: 事实 #3, #5, #6, #7, + 推测 #2, #3。

### 6. 产品架构、workspace/default profile、当前项目开发流程的切分

1. **产品架构**在本草案中回答的是：
   - 产品为什么需要四层分离
   - 每层各自存什么、不存什么
   - 真理源地位、可变性、生命周期、可重建性如何区分
   - 各层如何共同服务因果调查主链  
   依据: 事实 #3, #5, #6, #7, #8。
2. **workspace/default profile**后续应回答的是：
   - 某个默认本地部署如何把四层映射到具体目录、数据库、对象存储和索引介质
   - 默认保留策略、默认 artifact layout、默认 source roots 是什么
   - 某个 profile 的本地优先实现如何折中简化  
   依据: 事实 #5, #6, #8, + 推测 #4, #5。
3. **当前项目开发流程与治理规则**回答的是：
   - 分支纪律与合流门禁
   - 预写入检查
   - 受保护文档门禁
   - worktree / session 纪律
   - 格式化策略  
   这些都属于仓库工程治理，不属于产品存储架构本体。依据: 事实 #8。
4. **本草案的明确边界**：本草案只定义产品存储职责，不给出任何具体 workspace 的目录映射表，也不吸收任何具体仓库开发纪律作为产品内核规则。依据: 事实 #8。

### 7. 具体 workspace 现实应如何被对待

1. 任何具体 workspace 的路径、目录、日志布局、临时运行面和内部工具状态，首先都属于**workspace reality / 过渡实现 / 仓库自用治理资产**，而不是产品存储架构的定义条文。依据: 事实 #8。
2. 这些具体现实后续可以被整理进单独的 `workspace profile` 文档，用来解释“某个实现如何映射产品四层”；但这一步应在产品职责层定稿之后再做。依据: 事实 #8, + 推测 #4。
3. 当前不应在本草案中逐条把任何具体路径硬指定到某一层，因为那会把“某个 workspace 怎么长”偷换成“产品必须怎么长”。依据: 事实 #5, #8。

### 8. 迁移原则

1. **先定义，再映射。** 先固定四层职责与边界，再定义 default profile，最后才处理具体 workspace 的现实映射与迁移。依据: 事实 #8, + 推测 #1, #4。
2. **先语义，再介质。** 先确定对象、关系、证据、run 和索引的存储语义，再讨论 SQLite、文件系统、对象存储或图数据库等具体实现。依据: 事实 #6, #7。
3. **先分真理源，再谈重建。** 先说清哪些层不可替代、哪些层可重建，才能谈备份、恢复、缓存与重建任务。依据: 事实 #6, #7。
4. **先切边界，再谈治理。** 产品存储架构定完之后，workspace/default profile 和仓库开发规则再分别成文，不应继续混写。依据: 事实 #8。

### 9. 当前明确不做什么

1. 本草案**不**把任何具体 workspace 的路径、目录、文件名当作产品级四层定义。依据: 事实 #5, #8。
2. 本草案**不**把任何仓库治理门禁、起草流程纪律或开发规则混入产品存储架构本体。依据: 事实 #8。
3. 本草案**不**冻结具体物理后端、具体表结构、具体 ID 编码规则；这些仍需后续对象模型、证据策略与 profile 文档继续收口。依据: 事实 #5, #6, #7, + 推测 #5。
4. 本草案**不**把产品重新定义成 AI SRE、通用 RAG、通用知识库或通用调度器。依据: 事实 #1, #2, #9。
