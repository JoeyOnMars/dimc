# 存储架构草案 v1（设计记录）

> **状态**: `历史设计记录`
> **角色**: 保留 2026-03 中期“先起草、后并入正式文档”的设计演化痕迹
> **当前真理源**: [docs/STORAGE_ARCHITECTURE.md](../STORAGE_ARCHITECTURE.md)

## 文档定位

1. 本文保留的是 `2026-03` 期间围绕 DIMCAUSE 存储架构做收敛时的设计稿历史。
2. 自 `2026-03-19` 起，本文**不再**是第一层正式共享入口，也**不再**承担产品级存储职责的上位真理源角色。
3. 当前正式有效的产品级存储定义，统一以 [docs/STORAGE_ARCHITECTURE.md](../STORAGE_ARCHITECTURE.md) 为准。
4. 本文仍保留在 `docs/PROPOSALS/`，仅用于：
   - 追溯当时为什么要从“路径/实现现实”回到“职责/真理源/生命周期”；
   - 说明哪些约束曾先在草案中被显式说清；
   - 为后续审计或复盘提供设计演化证据。

## 已收口到正式文档的关键约束

下列内容原先在本草案中被更集中地表达，现已并入正式存储架构文档：

1. Evidence Layer 内部至少分为：
   - `Source Materials`
   - `Generated Evidence Artifacts`
2. 两类 Evidence 子对象必须在 provenance、生命周期、版本语义和引用方式上显式区分。
3. Knowledge Layer 中的关系状态历史必须可回放。
4. “当前状态”只能是投影，不得成为唯一记录。
5. 四层差异首先取决于职责、真理源地位、可变性、生命周期与可重建性，而不是某个当前介质或目录。

这些约束现在都应从 [docs/STORAGE_ARCHITECTURE.md](../STORAGE_ARCHITECTURE.md) 读取，而不是从本文读取。

## 保留本文而不直接删除的原因

1. 本文不是纯重复垃圾；它真实记录了“正式文档尚未完全升格前”的一轮设计收敛过程。
2. 直接删除会抹掉一段重要的设计演化证据，后续难以解释为什么会出现“正式文档与 draft 并存”的阶段。
3. 但继续把本文当正式入口会制造双重真理源，所以只能降位为设计记录，不能再与正式文档并列。

## 当前使用规则

1. 需要判断当前产品级存储职责时，只读 [docs/STORAGE_ARCHITECTURE.md](../STORAGE_ARCHITECTURE.md)。
2. 需要研究历史设计思路、审计设计演化或复盘文档治理问题时，才回看本文。
3. 新的 proposal、正式文档和实现说明，不应再把本文当作产品级事实依据直接引用。
