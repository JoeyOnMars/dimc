## 一、背景与目标

**现状问题**：
- `dimc down` 只将原始 Claude JSONL 复制进 `end.md`，无结构化事件提取
- `run_smart_scan()` 完全依赖 `DEEPSEEK_API_KEY`，本地无法运行

**目标**：三层提取管道

```
session_end.py
  └── ExtractionPipeline.run(session_data)
        ├── L0 Ingest:   原始文本 → chunks.db (status='raw')
        ├── L1 Embed:    BGE-M3（离线，始终运行）→ 嵌入向量
        ├── L1 Extract:  SessionExtractor → L1 事件（低置信度）
        ├── L2 Extract:  [可选] 云端 LLM → L2 事件（高置信度）
        ├── Validate:    Pydantic ExtractedEventRecord 校验门
        ├── Persist:     EventIndex.add() → events 表
        └── _link_causal_edges() → GraphStore 因果边
```

**FSM 状态机**（chunks 表）：

```
  raw ──embed──► embedded ──extract──► extracted
         │                    │
      [fail]               [fail]
      retry_count++        retry_count++
      if >= MAX_RETRIES:   if >= MAX_RETRIES:
        extraction_failed=True, needs_extraction=False
```

## 二、核心架构决策（Option C）

**L1/L2 共存，读时 COALESCE**：
- L1 和 L2 各自独立生成事件，不在写入时强行对应
- 查询时通过 UNION ALL 取最优（L2 优先于 L1，历史事件走独立路径）
- 避免了软替换（`superseded_by`）的写入竞争和幽灵事件问题

**L2 重试幂等性**：
- L2 重试前先 DELETE 该 chunk 全部旧 L2 事件，再 INSERT 新批次（单库事务）
- 防止失败重试留下幽灵事件

## 三、26 条设计定案（全部封闭）

| # | 定案 |
|---|------|
| 1 | `chunk_id = "chk_" + sha256(f"{source_event_id}:{content[:64]}").hexdigest()[:16]` |
| 2 | chunks.db 独立文件（`~/.dimcause/chunks.db`），不放进 index.db |
| 3 | `session_id` 为 chunks 表一级字段（非 metadata JSON） |
| 4 | `event_ids` 为 chunks 表 TEXT 字段，存 JSON 数组（`[]` 初始值） |
| 5 | L1/L2 event_id 命名空间隔离：`sha256(chunk_id:l1:index)` vs `sha256(chunk_id:l2:index)` |
| 6 | TLS 连接池：`threading.local()` + `PRAGMA busy_timeout=5000` |
| 7 | `embedding_version` 和 `extraction_version` 各为独立整数字段 |
| 8 | events 表不加 `superseded_by`（Option C 完全取代 M-4） |
| 9 | events 表新增 `source_chunk_id TEXT DEFAULT NULL` |
| 10 | events 表新增 `source_layer TEXT DEFAULT NULL CHECK(IS NULL OR IN ('l1','l2'))` |
| 11 | L1 写策略：`INSERT OR IGNORE`（重试幂等） |
| 12 | L2 写策略：group DELETE + INSERT 单库事务 |
| 13 | CausalLinker 代表事件：per-chunk per-type（每个 chunk 每种 event_type 取一个代表） |
| 14 | COALESCE 查询：UNION ALL 两路（路径 A：pipeline events；路径 B：历史 events） |
| 15 | 跨库写入顺序：先写 event_index.db，成功后再更新 chunks.db |
| 16 | `event_ids` 字段更新封装在 `ChunkStore` 内部 |
| 17 | L1 GC 推迟至 V6.4 |
| 18 | COALESCE 过滤机制：EXISTS/NOT EXISTS（非 GROUP BY + MAX(updated_at)） |
| 19 | `source_chunk_id IS NULL` 的历史事件走路径 B，独立于路径 A |
| 20 | CausalLinker 粒度：per-chunk per-type |
| 21 | `idx_chunk_layer`：`ON events(source_chunk_id, source_layer, updated_at DESC)` |
| 22 | CHECK CONSTRAINT NULL 安全写法：`IS NULL OR source_layer IN ('l1', 'l2')` |
| 23 | 所有含 NULL 合法值的枚举约束均采用 NULL 安全写法 |
| 24 | `graph_edges` 字段名：`source`/`target`/`relation` |
| 25 | `add_relation()` 策略：**RMW + BEGIN IMMEDIATE**（weight 取 MAX，metadata 列表去重追加，`created_at` 首次写入后不变） |
| 26 | CausalLinker 采用外键校验建边：事件入库后遍历 `related_event_ids`，经 ontology 校验后同步写边；N×M 增量扫描系统全部废弃 |

## 五、关键查询 SQL

### 5.1 COALESCE 读时决战（定案 14、18、19）

```sql
-- 路径 A：流水线事件，L2 优先于 L1
SELECT e.* FROM events e
WHERE e.source_chunk_id IS NOT NULL
  AND (
    (e.source_layer = 'l2' AND EXISTS (
        SELECT 1 FROM events e2
        WHERE e2.source_chunk_id = e.source_chunk_id
          AND e2.source_layer = 'l2'))
    OR
    (e.source_layer = 'l1' AND NOT EXISTS (
        SELECT 1 FROM events e2
        WHERE e2.source_chunk_id = e.source_chunk_id
          AND e2.source_layer = 'l2'))
  )
UNION ALL
-- 路径 B：历史事件（无 chunk 来源），全量返回
SELECT e.* FROM events e
WHERE e.source_chunk_id IS NULL;
```

### 5.2 CausalLinker 代表事件查询（定案 13、20）

```sql
-- per-chunk per-type：每个 chunk 每种 event_type 取代表（L2 > L1）
SELECT e.* FROM events e
WHERE e.source_chunk_id IN (/* 本 session 的 chunk_id 列表 */)
  AND (
    (e.source_layer = 'l2' AND EXISTS (
        SELECT 1 FROM events e2
        WHERE e2.source_chunk_id = e.source_chunk_id
          AND e2.type = e.type
          AND e2.source_layer = 'l2'))
    OR
    (e.source_layer = 'l1' AND NOT EXISTS (
        SELECT 1 FROM events e2
        WHERE e2.source_chunk_id = e.source_chunk_id
          AND e2.type = e.type
          AND e2.source_layer = 'l2'))
  )
```

## 六、关键实现契约

### 6.1 `add_relation()` RMW 实现（定案 25）

**设计理由**：

| 策略 | 问题 |
|------|------|
| `INSERT OR IGNORE` | 旧 weight 永久锁死，L2 改进被静默丢弃 |
| `INSERT OR REPLACE` | 物理 DELETE+INSERT，metadata 被完全覆盖，历史来源永久丢失 |
| `ON CONFLICT DO UPDATE + json_patch` | json_patch 是数组替换（RFC 6902 replace），非追加合并 |
| **RMW + BEGIN IMMEDIATE** | ✓ weight 取 MAX，metadata 列表去重追加，created_at 首次写入后不变 |

**前提**：`_get_conn()` 必须设 `isolation_level=None`，否则 Python 隐式 BEGIN 与 BEGIN IMMEDIATE 产生嵌套事务错误。

```python
# _get_conn() 最小改动（仅加一行）
def _get_conn(self) -> sqlite3.Connection:
    conn = sqlite3.connect(str(self.db_path), timeout=10.0)
    conn.isolation_level = None   # autocommit，手动控制事务
    conn.row_factory = sqlite3.Row
    return conn
```

```python
def add_relation(
    self,
    source: str,
    target: str,
    relation: str,
    weight: float = 1.0,
    metadata: dict | None = None,
) -> None:
    """
    插入或条件更新因果边。
    策略：RMW + BEGIN IMMEDIATE（行锁，防并发竞争）
    - weight：取 MAX（置信度只升不降；V6.4 可改为加权平均）
    - metadata：列表字段去重追加，标量字段新值覆盖
    - created_at：首次 INSERT 时记录，UPDATE 时不动
    """
    metadata = metadata or {}
    conn = self._get_conn()
    conn.execute("BEGIN IMMEDIATE")
    try:
        row = conn.execute(
            "SELECT weight, metadata FROM graph_edges "
            "WHERE source=? AND target=? AND relation=?",
            (source, target, relation)
        ).fetchone()
        if row:
            new_weight = max(row[0], weight)
            new_meta = _merge_metadata(json.loads(row[1] or '{}'), metadata)
            conn.execute(
                "UPDATE graph_edges SET weight=?, metadata=? "
                "WHERE source=? AND target=? AND relation=?",
                (new_weight, json.dumps(new_meta), source, target, relation)
            )
        else:
            conn.execute(
                "INSERT INTO graph_edges "
                "(source, target, relation, weight, metadata, created_at) "
                "VALUES (?, ?, ?, ?, ?, ?)",
                (source, target, relation,
                 weight, json.dumps(metadata), time.time())
            )
        conn.execute("COMMIT")
    except Exception:
        conn.execute("ROLLBACK")
        raise
    finally:
        conn.close()


def _merge_metadata(old: dict, new: dict) -> dict:
    """列表字段去重追加（旧值在前），标量字段新值覆盖"""
    result = dict(old)
    for k, v in new.items():
        if isinstance(v, list) and isinstance(result.get(k), list):
            seen, merged = set(), []
            for item in result[k] + v:
                key = (item if not isinstance(item, dict)
                       else json.dumps(item, sort_keys=True))
                if key not in seen:
                    seen.add(key)
                    merged.append(item)
            result[k] = merged
        else:
            result[k] = v
    return result
```

> **注意**：`_upsert_edge()`（供 `add_event_relations()` 使用）保持不变，它在调用方已持有连接和事务的上下文中运行，与 `add_relation()` 的独立 RMW 逻辑分离。

## 十一、CausalLinker 架构演进记录（ADR）

> **v1.0**（管道内单 chunk 绑定）→ **v2.0**（N×M 增量扫描）→ **v3.0（当前）**（外键校验，N×M 全部废弃）

### 废弃路径与根因

| 方案 | 废弃原因 |
|------|---------|
| 管道内单 chunk 绑定（v1.0） | 跨 chunk 因果边永久断裂 |
| 全 session N×N | O(N²) 在 2000+ 事件 session 下卡死 CLI |
| N×M 增量扫描（v2.0，含 linker_queue / 视野池） | 漏洞 9：纯类型匹配产生笛卡尔积爆炸（N×M 废边）；漏洞 11：cos_sim ≠ causality，向量门控算法基础不成立；漏洞 13：embedding 粒度在 chunk 层，event 层向量不存在 |
| 向量相似度门控（M 专家定案 33 草案） | 同上（漏洞 11、13），彻底否定 |

### 当前采用方案（v3.0）：外键校验建边

**原则**：V6.3 offline-first 约束下，不引入 Causal NLI，不做自动因果推断。建边能力完全依赖 L2 大模型在提取时的显式外键声明。

**建边逻辑（ExtractionPipeline._persist_events() 内同步执行）**：

```python
def _link_causal_edges(self, event: Event) -> None:
    # Event 经 Pydantic model_validate 反序列化，related_event_ids 已是 Python list，
    # 不得套 json.loads()；isinstance 兜底处理直接从 DB row 构建的边缘路径。
    related_ids = event.related_event_ids or []
    if isinstance(related_ids, str):
        related_ids = json.loads(related_ids)
    if not related_ids:
        return
    for target_id in related_ids:
        target = self.event_index.get(target_id)
        if target is None:
            logger.warning(f"[CausalLinker] 目标事件不存在: {target_id}，跳过")
            continue
        # 本体关系有方向性（domain → range）；声明引用方未必是 source，
        # 需双向搜索本体，动态决定 source/target（详见 V6.3_CAUSAL_LINKER_DESIGN.md 第三节）。
        result = _infer_directed_relation(event, target)
        if result is None:
            logger.warning(
                f"[CausalLinker] 本体中无合法关系: {event.type} ↔ {target.type}，跳过"
            )
            continue
        relation, source_evt, target_evt = result
        valid, msg = self.ontology.validate_relation(
            relation, source_evt.type, target_evt.type
        )
        if not valid:
            logger.warning(f"[CausalLinker] 本体校验失败: {msg}，跳过")
            continue
        self.graph_store.add_relation(
            source=source_evt.id,
            target=target_evt.id,
            relation=relation,
            weight=1.0,
            metadata={"derived_from_chunks": [event.source_chunk_id]},
        )
```

**约束**：
- L2 大模型在原文有明确上下文引用时填充 `related_event_ids`，无明确引用返回 `[]`，**不得幻构**
- 所有外键路径必须经过 `OntologyEngine.validate_relation()` 校验，没有例外
- 无外键事件作为孤岛存在，不再寻找前世今生
