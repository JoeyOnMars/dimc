# Evidence Policy and Causality Grades v1

## 文档定位

1. 本文是**产品职责层**的《Evidence Policy and Causality Grades v1》草案，目标是先定义产品级证据政策、证据覆盖等级与关系确定性等级，再作为后续正式架构文档与领域 profile 的上位约束。
2. 本文**不是**当前 repo 的实现说明，不是当前 workspace 的目录或数据库映射，也不是当前开发流程规则汇总。
3. 本文只定义产品为什么需要两组等级、等级分别表达什么、哪些证据足以进入某一级、缺失与冲突如何表达，以及这些规则如何与对象模型和四层存储对齐。
4. 本文**不**冻结数据库表结构、目录布局、ID 编码规则、默认本地路径或任何当前实现 profile 的物理细节。

## 已知事实

1. 当前正式产品架构已经收口为：**DIMCAUSE 是一套面向本地异构材料的证据驱动因果调查系统。**依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)；[WORKSPACE_PROFILE_V1.md](./WORKSPACE_PROFILE_V1.md)。
2. 产品核心目标不是通用知识库、通用 RAG 平台或通用开发调度器，而是从给定材料中抽取、验证、组织尽可能强的因果关系，并输出可追溯、可分级、可解释的调查结论。依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)。
3. 当前正式产品架构已经把系统主线收敛为：接收本地材料、提升为结构化对象、生成并验证因果关系、输出带证据链和等级的解释结论。依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)。
4. 正式产品架构已经明确要求把两组分级拆开：证据覆盖等级 `E1-E4`，关系确定性等级 `C0-C4`；并要求用户界面同时展示当前等级、主要证据来源、缺失证据说明与等级历史。依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)。
5. 当前正式产品架构已经明确了两组等级的职责分工：`E` 回答“当前证据面有多完整”，`C` 回答“在这些证据下，这条关系有多能成立”；两组等级必须同时展示，不能压成单一总分。依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)。
6. 正式产品架构已经明确：Layer 3 不是单纯 ontology，而是 `Ontology and Evidence Policy`；它需要同时约束对象类型、关系类型、证据政策、等级表达以及缺失/反证处理。依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)。
7. 当前正式产品架构与正式存储架构已经明确三项根本判断：统一运行单位应是 `Run`；系统要从开发材料偏置升级到通用本地材料对象化与证据化因果推理；存储要切成 `Evidence / Runtime / Knowledge / Derived Index` 四层。依据：[PROJECT_ARCHITECTURE.md](../PROJECT_ARCHITECTURE.md)；[STORAGE_ARCHITECTURE.md](../STORAGE_ARCHITECTURE.md)。
8. 当前正式存储架构已经要求：
   - `Evidence Layer` 保存 `Source Materials` 与 `Generated Evidence Artifacts`
   - `Knowledge Layer` 保存对象、关系、证据绑定、等级和关系状态历史
   - “当前状态”只是投影，不得成为唯一记录  
   依据：[STORAGE_ARCHITECTURE.md](../STORAGE_ARCHITECTURE.md)。
9. 当前《Core Object Model v1》已经把 `Run`、`Material`、`MaterialVersion`、`Entity`、`Event`、`Decision`、`Claim`、`Task`、`Symbol`、`Artifact`、`Check`、`Result`、`Relation` 定义为产品通用内核的一等对象家族，并明确：
   - `Run` 是 `Runtime-first`
   - `Material / MaterialVersion` 是 `Knowledge + Evidence` 双重对象
   - `Claim != Relation`
   - `Check != Result`
   - `Artifact != Generated Evidence Artifacts`  
   依据：[CORE_OBJECT_MODEL_V1.md](./CORE_OBJECT_MODEL_V1.md)。
10. 当前正式共享文档已经明确：产品架构、workspace/default profile、当前项目开发流程必须切开；任何内部治理规则与具体 workspace 现实都不应混入产品架构正文。依据：[ARCHITECTURE_INDEX.md](../ARCHITECTURE_INDEX.md)；[WORKSPACE_PROFILE_V1.md](./WORKSPACE_PROFILE_V1.md)；[REPO_WORKFLOW_AND_GOVERNANCE_BOUNDARY_V1.md](./REPO_WORKFLOW_AND_GOVERNANCE_BOUNDARY_V1.md)。

## 推测与假设

1. [推测] 这份文档的主语应当是**关系与命题的证据化判断规则**，而不是对所有对象一律打分。原因：事实 #2、#3、#4、#9 说明产品最终要输出的是因果关系与解释，而不是把所有对象都变成“评分卡”。
2. [推测] `Relation` 应是两组等级的主要承载对象，`Claim` 应是次级承载对象。原因：`Claim` 可以作为命题层候选，`Relation` 则是进入对象网络后的结构化边；二者需要共用一套证据政策，但承担不同阶段职责。[基于事实 #3, #8, #9]
3. [推测] 证据覆盖等级与关系确定性等级必须保持**可组合但不互相替代**：高覆盖不等于高确定性，高确定性也不等于高覆盖。原因：事实 #4、#5 已经先把两组等级拆开；如果后续又把它们重新压成单一分数，就会失去产品展示与调查约束的意义。
4. [推测] `Run`、`Check`、`Result`、`Generated Evidence Artifacts` 等对象，更多承担 provenance、验证和回放责任，而不是直接成为“因果真理源”。原因：事实 #6、#7、#8、#9 已经把 Evidence、Runtime、Knowledge 的职责切开。
5. [推测] “缺失证据”与“冲突证据”都必须进入 Knowledge 语义，而不能只作为解释文案临时生成。原因：事实 #4、#8 已经要求等级、状态历史和证据绑定进入 Knowledge。
6. [推测] 两组等级本身也应具有历史，因为如果关系状态历史必须可回放，那么等级变化和等级变化依据也必须可追踪。原因：事实 #8 已明确要求“当前状态只是投影”。
7. [推测] 领域 profile 后续可以细化“什么算直接证据、什么算强证据、哪些源更权威”，但产品通用内核必须先把等级语义、缺失/冲突表达、历史保留原则立住。原因：事实 #6、#9、#10 已经要求产品内核和领域 profile 分离。

## 结论与建议

### 1. 总体判断

1. 《Evidence Policy and Causality Grades v1》应被定义为：**为证据驱动因果调查系统提供一套双轴判断规则，使系统能同时表达“证据面有多完整”与“关系在这些证据下有多站得住”，并把缺失、反证、历史变化纳入同一套产品语义。**依据: 事实 #1, #2, #3, #4, #5, #6, #8 + 推测 #1, #2, #3, #5, #6。
2. 这份文档的主语必须是**产品级证据政策**，不是某个当前实现的评分公式，也不是某个当前 workspace 的文件落盘规范。依据: 事实 #7, #8, #9, #10 + 推测 #4, #7。
3. 这份文档要解决的是“何时可以说支持、何时只能说候选、何时必须降级、何时必须显式承认缺口或反证”；它不负责定义物理 schema 或具体目录。依据: 事实 #4, #6, #10 + 推测 #3, #5, #7。

### 2. 两组等级的切分原则

1. **证据覆盖等级 `E` 回答的是：当前被纳入评估范围的证据面有多完整。** 它表达的是调查覆盖面、时间跨度、协作上下文和过程可回放程度，而不是关系是否为真。依据: 事实 #4, #5 + 推测 #3。
2. **关系确定性等级 `C` 回答的是：在当前可用证据下，这条 `Claim` 或 `Relation` 有多能成立。** 它表达的是支持强度、直接性、独立性、关键链路闭合程度，以及反证是否被解决。依据: 事实 #4, #5 + 推测 #2, #3。
3. 两组等级必须同时展示，不允许被压成单一总分。依据: 事实 #4, #5 + 推测 #3。
4. 允许存在 `E` 高而 `C` 低，也允许存在 `E` 低而 `C` 高：
   - `E4 / C0` 代表证据面很全，但冲突或反证使当前结论站不住。
   - `E1 / C4` 代表当前快照中存在直接强证据，但证据面仍不完整。  
   这不是异常，而是双轴模型的设计目的。依据: 事实 #4, #5 + 推测 #3。
5. 任何高等级都不应隐藏缺失项与反证；等级越高，越需要说明依据和边界。依据: 事实 #4 + 推测 #5。
6. 本稿首先规范的是 `Claim / Relation` 级别的 `E` 与 `C`。调查级、报告级或 run 级的整体 `E` 只能作为聚合投影，用于概览当前调查覆盖面；它不得替代单条 `Claim / Relation` 的等级判断，也不得反向覆盖对象级评估。依据: 事实 #4, #5, #8, #9 + 推测 #3, #6。

### 3. 等级的主要承载对象

1. `Relation` 是两组等级的**主要承载对象**。产品最终要解释的是对象之间哪些关系成立、哪些只是候选、哪些被否决。依据: 事实 #2, #3, #8, #9 + 推测 #2。
2. `Claim` 是两组等级的**次级承载对象**。当命题尚未整理成稳定 `Relation` 时，系统仍应能对其做 provisional assessment。依据: 事实 #3, #9 + 推测 #1, #2。
3. `Check` 与 `Result` 不是最终因果等级的替代物，但它们是提升、维持、降低或否定某个 `Claim / Relation` 的关键证据输入。依据: 事实 #5, #6, #9 + 推测 #4。
4. `Run` 不是因果真理本身，但它是 `E4 Full Evidence` 里“过程可回放”的关键 provenance 对象。依据: 事实 #5, #7, #8, #9 + 推测 #4。
5. `Material / MaterialVersion` 是 `E1` 与 `E2` 的基础锚点；没有材料快照与版本历史，就不可能谈快照级或历史级覆盖。依据: 事实 #3, #5, #8, #9。
6. `Entity`、`Event`、`Decision`、`Task`、`Symbol`、`Artifact` 是被 `Relation` 连接、被 `Claim` 指涉、被 `Check / Result` 验证的主要对象端点与上下文对象。依据: 事实 #3, #9。

### 4. 证据覆盖等级 `E1-E4`

| 等级 | 含义 | 进入该等级至少需要 | 典型缺口 |
| --- | --- | --- | --- |
| `E1 Snapshot` | 只有当前材料快照层面的覆盖 | 至少有一份与该 `Claim / Relation` 直接绑定的 `MaterialVersion`、原始输入快照或其他当前时点证据载体 | 缺少历史演化、协作上下文、过程证据 |
| `E2 Historical` | 已能看到时间或版本演化 | 满足 `E1`，且至少存在一条可靠的历史链：版本历史、时间锚点序列、前后状态对照或可验证的演化记录 | 缺少协作决策、评审、检查或过程性上下文 |
| `E3 Collaborative` | 已能看到跨角色或跨环节协作证据 | 满足 `E2`，且存在与该判断直接相关的协作材料，如讨论、评审、Issue、PR、Decision、Check、外部交互记录等 | 缺少本地运行与调查过程的完整回放链 |
| `E4 Full Evidence` | 已能回放相对完整的调查或执行过程 | 满足 `E3`，且存在可回放的过程证据链，例如 `Task / Run / Discussion / Check / Result / Artifact` 等对象或工件能把输入、判断、验证、产出串成闭环 | 不要求“所有世界信息都齐全”，但要求当前调查范围内的关键过程证据闭环已形成 |

1. `E` 等级衡量的是**覆盖面**，不是结论强度。达到 `E4` 不代表某关系一定成立，只代表当前调查证据面相对完整。依据: 事实 #4, #5 + 推测 #3。
2. `E2` 的核心不是“材料变多了”，而是“已经有时间或版本演化的证据”。如果只有多个当前快照，但没有可成立的历史链，仍不应高于 `E1`。依据: 事实 #5。
3. `E3` 的核心不是“来源数量更多”，而是“出现了与判断直接相关的跨角色、跨环节协作上下文”。依据: 事实 #5 + 推测 #7。
4. `E4` 的核心不是“所有材料都导进来了”，而是“在当前调查范围内，系统已经具备相对完整的过程证据闭环，可用于回放、审计和解释”。依据: 事实 #5, #7, #8, #9 + 推测 #4。
5. `Run`、`Check`、`Result`、`Generated Evidence Artifacts` 会显著影响 `E4`，但它们不应被误写成 `Knowledge` 之外的唯一评分真理源。依据: 事实 #8, #9 + 推测 #4。
6. `E4` 的核心标准是：在当前调查范围内，关键过程证据闭环已经形成。上文列出的 `Task / Run / Discussion / Check / Result / Artifact` 只是工程 profile 的示例对象链，不得被解释为所有领域进入 `E4` 的固定必备对象集合。依据: 事实 #5, #7, #8, #9 + 推测 #4, #7。

### 5. 关系确定性等级 `C0-C4`

| 等级 | 含义 | 进入该等级至少需要 | 当前不应宣称什么 |
| --- | --- | --- | --- |
| `C4 Confirmed` | 当前范围内有明确、直接、强且未被实质反驳的证据支持该关系 | 关键端点已绑定，关键证据直接关联该关系，关键链路基本闭合，且不存在未解决的实质反证 | 不应把它表述成超出当前调查范围的绝对真理 |
| `C3 Strongly Supported` | 有多条强证据共同支持，但仍缺少某个直接闭环或最终锚点 | 至少有两类以上相互强化的强证据，且没有足以推翻它的未解决反证 | 不应把它说成“已经直接证实” |
| `C2 Supported` | 有合理支持，但仍有关键缺口 | 已有一定证据支持方向性、关联性或局部机制，但至少存在一个关键链路未闭合 | 不应把它当作最终因果结论 |
| `C1 Hypothesized` | 主要是候选推断 | 只有弱证据、间接线索、局部共现或尚未验证的命题支撑 | 不应被当作已支持关系对外输出 |
| `C0 Contradicted or Insufficient` | 当前要么证据不足，要么存在实质反证 | 当前证据不足以超过 `C1`，或已有关键反证/冲突证据尚未解决 | 不应继续被表述成当前成立的关系 |

1. `C` 等级衡量的是**在当前证据下的可成立程度**，不是材料覆盖面。依据: 事实 #4, #5 + 推测 #3。
2. `C4` 需要的是“当前范围内的直接强支持”，而不是“宇宙级完备覆盖”；因此 `C4` 理论上可以与 `E1`、`E2` 组合出现。依据: 事实 #4, #5 + 推测 #3。
3. `C3` 与 `C2` 的主要差别，不是“证据数量多少”，而是**关键链路是否已大致闭合、反证是否已被压住**。依据: 事实 #5 + 推测 #3, #5。
4. `C1` 允许系统保留调查候选，但必须显式承认这仍是推断，不得伪装成已验证知识。依据: 事实 #5。
5. `C0` 不是“没有记录就算了”，而是一个必须被明确表达的状态：它要么表示证据不足，要么表示当前存在未解决反证。依据: 事实 #4, #5 + 推测 #5。
6. 任何被标记为 `C0` 的 `Claim / Relation`，都必须同时携带不可省略的原因子类，至少区分为 `contradicted` 与 `insufficient`。系统不得只保存一个无说明的 `C0`。依据: 事实 #5, #8, #9 + 推测 #5, #6。

### 6. 缺失证据如何表达

1. 缺失证据必须作为**一等调查信息**进入产品语义，不能只在解释文案里临时补一句“可能还缺一些材料”。依据: 事实 #4, #8 + 推测 #5。
2. 对每个被展示的 `Claim / Relation`，系统至少应能表达四类缺失信息：
   - 当前缺的是哪一类证据
   - 该缺口阻止了哪一级提升
   - 缺口对应的是哪个对象、时间窗或过程环节
   - 缺口是“未导入”“未提取”“未验证”还是“理论上应有但目前不可得”  
   依据: 事实 #4, #6 + 推测 #5, #7。
3. 缺失证据的表达必须与当前等级绑定：例如某关系之所以停在 `C2`，必须能解释是哪个关键证据缺口没有闭合。依据: 事实 #4, #5 + 推测 #5。
4. 对 `E` 等级，缺失证据主要回答“覆盖面还差什么”；对 `C` 等级，缺失证据主要回答“为什么这条关系还不能更强”。依据: 事实 #4, #5 + 推测 #3, #5。

### 7. 反证与冲突证据如何处理

1. 反证与冲突证据必须被显式保留、显式绑定、显式展示；它们不是噪声，也不能因为不利于当前结论就被覆盖掉。依据: 事实 #2, #4, #8 + 推测 #5, #6。
2. 冲突证据对两组等级的影响不同：
   - 它可能**提高 `E`**，因为调查覆盖面变得更完整。
   - 它也可能**降低 `C`**，因为当前关系的可成立程度被削弱。  
   依据: 事实 #4, #5 + 推测 #3, #5。
3. 如果存在未解决的关键反证，关系不应进入 `C4`；如果该反证已足以否定当前表述，关系应落到 `C0`。依据: 事实 #5 + 推测 #5。
4. “被否决关系”与“有反证但尚未定论的关系”必须区分：
   - 前者是当前判断已明确不成立
   - 后者是当前仍在争议或仍有待验证  
   依据: 事实 #8, #9 + 推测 #2, #5。
5. 冲突解决过程本身也应成为历史的一部分；后续用户需要看到“为什么原来是 `C3`，后来变成 `C0`”。依据: 事实 #8 + 推测 #6。

### 8. 当前状态与历史状态

1. 任一 `Claim / Relation` 的等级都应被理解为：**某一时点、某一范围、某一证据集下的评估结果**。依据: 事实 #4, #8 + 推测 #6。
2. “当前等级”只能是最新投影，不能成为唯一记录。等级变化、变化依据、变化时间和变化前后的证据差异都应可回放。依据: 事实 #8 + 推测 #6。
3. 历史状态对 `E` 的影响至少有两点：
   - 没有历史链，就不应超过 `E1`
   - 新增历史证据后，`E` 可以上升；若旧证据后来被判定不相关或失效，当前 `E` 可以下降，但历史判断不得被覆盖抹除  
   依据: 事实 #5, #8 + 推测 #6。
4. 历史状态对 `C` 的影响至少有两点：
   - 后续新增支持证据可以让关系从 `C1/C2` 上升到 `C3/C4`
   - 后续新增反证也可以让关系从 `C3/C4` 降到 `C2/C1/C0`  
   依据: 事实 #5, #8 + 推测 #5, #6。
5. 因果调查系统必须支持用户追问：“这条关系今天为什么是 `C2`，上周为什么是 `C1`，中间新增了哪些证据或反证？” 这正是等级历史必须保留的原因。依据: 事实 #2, #4, #8 + 推测 #6。

### 9. 与对象模型的对齐

| 对象 | 在证据政策中的主要职责 | 是否为等级主承载对象 |
| --- | --- | --- |
| `Run` | 提供过程 provenance、生成链与 `E4` 所需的运行回放线索 | 否 |
| `Material` | 提供材料身份与调查范围边界 | 否 |
| `MaterialVersion` | 提供快照、版本与时间锚点，是 `E1/E2` 的基础 | 否 |
| `Entity` | 作为关系端点或命题主语之一 | 否 |
| `Event` | 提供时间上可锚定的发生与变化对象 | 否 |
| `Decision` | 提供可影响因果判断的承诺性选择证据 | 否 |
| `Claim` | 命题层评估对象，可带 provisional `E/C` | 是，次级 |
| `Task` | 提供过程目标、执行语境与 `E4` 的一部分 | 否 |
| `Symbol` | 提供可寻址的形式化对象端点 | 否 |
| `Artifact` | 作为对象层工件参与关系与命题 | 否 |
| `Check` | 提供验证动作或验证义务 | 否 |
| `Result` | 提供结构化验证结果与支持/反证信号 | 否 |
| `Relation` | 关系级证据绑定、状态历史、等级历史的主要宿主 | 是，主要 |

1. `Claim` 与 `Relation` 共用同一套等级语义，但 `Relation` 的等级更接近最终解释输出，`Claim` 的等级更接近调查中间态。依据: 事实 #3, #8, #9 + 推测 #1, #2。
2. `Check / Result` 必须能同时为“支持”与“反证”服务，不能被预设为只会提升等级。依据: 事实 #5, #6, #9 + 推测 #5。
3. `Artifact` 与 `Generated Evidence Artifacts` 仍然必须切开：前者是对象模型工件，后者是 Evidence Layer 里的证据工件。证据政策需要同时引用两者，但不能把它们混成一个对象。依据: 事实 #8, #9。

### 10. 与四层存储的对齐

1. **Evidence Layer** 保存支撑等级判断的原始证据和可审计工件，包括 `Source Materials`、`Generated Evidence Artifacts`、检查附件、报告工件和可回放片段。它回答“依据是什么”。依据: 事实 #7, #8。
2. **Runtime Layer** 保存正在进行中的评估 run、检查 run、验证 run 的可变状态。它可以承载临时评估过程，但不应成为最终等级真理源。依据: 事实 #7, #8, #9。
3. **Knowledge Layer** 保存 `Claim / Relation` 的当前等级、等级历史、支持证据绑定、反证绑定、缺失证据说明、状态迁移与解释元数据。它回答“当前系统如何判断，以及为什么”。依据: 事实 #7, #8, #9 + 推测 #5, #6。
4. **Derived Index Layer** 只保存为召回、排序、检索和图遍历服务的派生结构；它可以帮助发现证据，但不应承载唯一等级记录。依据: 事实 #7, #8。
5. 缺失证据本身不属于 Evidence Layer 的“某个文件”，但它必须作为 Knowledge 里的调查约束被持久表达，并可引用到“本应存在但当前缺失的是哪一类证据”。依据: 事实 #8 + 推测 #5。

### 11. 产品通用内核与领域 profile 的切分

1. 产品通用内核必须先固定以下内容：
   - `E1-E4` 与 `C0-C4` 的基本语义
   - 两组等级必须同时展示
   - 缺失证据与冲突证据必须显式表达
   - 当前等级只是投影，等级历史必须保留
   - 哪些对象主要提供证据，哪些对象主要承载等级  
   依据: 事实 #4, #5, #6, #8, #9, #10 + 推测 #6, #7。
2. 领域 profile 后续再细化以下内容：
   - 哪些源在该领域被视为“直接证据”或“强证据”
   - 哪些 `Relation` 类型在该领域允许进入 `C4`
   - 哪些 `Check / Result` 在该领域具有更高权重
   - 缺失证据分类与解释模板的领域化表达  
   依据: 事实 #6, #9, #10 + 推测 #7。
3. 领域 profile 可以细化阈值，但不应改变双轴模型、历史保留原则和缺失/反证显式化原则。依据: 事实 #6, #10 + 推测 #7。

### 12. 当前明确不做什么

1. 本稿**不**定义数据库表、目录结构、JSON schema、ID 编码规则或默认本地路径。依据: 事实 #10。
2. 本稿**不**把任何内部治理规则、当前 workspace 现实或实现映射写成证据政策的一部分。依据: 事实 #10。
3. 本稿**不**在本轮冻结所有领域的“直接证据”判据；这些需要后续由各领域 profile 在本内核之上细化。依据: 事实 #6, #10 + 推测 #7。
4. 本稿**不**把产品重新定义成 AI SRE、通用 RAG、通用知识库或通用 memory OS。依据: 事实 #1, #2。
